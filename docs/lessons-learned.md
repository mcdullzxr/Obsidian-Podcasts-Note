# 经验沉淀：从返工中学到的

> 归档时间：2026-04-17
> 上下文：MVP 开发期间踩了 4 个大坑（CORS、Whisper 413、LLM 404、MiniMax 协议不对）

---

## 一、本次返工回顾

| # | 问题 | 根因 | 浪费的工作量 |
|---|------|------|------------|
| 1 | `Failed to fetch` | 在 Obsidian 里用原生 `fetch` 会被 CORS 拦 | 1 次排查 + 1 次改造 |
| 2 | Whisper 413 | 用错了服务商（MiniMax 根本没有转录服务） | 从单 provider 重构为多 provider dispatcher |
| 3 | LLM 404 | MiniMax 端点路径不是 `/v1/chat/completions` | 加 `resolveChatEndpoint` 兜底 |
| 4 | MiniMax M2.7 还是 404 | M2.7 系列根本不走 OpenAI 协议，用 Anthropic 协议 | 再次重构，加协议字段 + 分支实现 |

**规律：四个坑都可以在编码前 30 分钟 API 调研里避免。**

## 二、可沉淀的工程经验

### 1. **API 调研前置化**（最重要）

动手写代码前，必须先完成一张「API 兼容矩阵」：

| 服务商 | 有转录吗 | 有 LLM 吗 | LLM 协议 | 端点格式 | 鉴权方式 | 关键坑 |
|--------|---------|----------|---------|---------|---------|-------|
| OpenAI | ✅ Whisper | ✅ | OpenAI | `/v1/chat/completions` | Bearer | — |
| MiniMax | ❌ | ✅ | **Anthropic**（M2.7）/ OpenAI（M2-her）| `/anthropic/v1/messages` | `x-api-key` | 协议/端点都和别人不一样 |
| 火山 SeedASR | ✅ | — | — | 异步 submit+query | `X-Api-Key` | 要轮询 |
| 通义 Paraformer | ✅ | — | — | 异步 submit+poll+fetch | Bearer | 三步走 |

**做完这张表再动手，不会出现 MVP 写完又返工的情况。**

### 2. **默认遵循"有抽象层"的架构**

多 provider 场景下，一开始就该用 dispatcher 模式：

```
dispatcher.ts  → 按 config.provider 分发
providers/
  ├─ openai.ts
  ├─ anthropic.ts
  └─ volcengine.ts
```

**不要先"写死一个 provider 再重构"**——前两次重构都是因为这个。

### 3. **BYOK 产品的铁律**

- 别假设"我的 key 能用 → 别人的 key 也能用"：套餐/协议/端点都可能不同
- UI 里要把**协议**独立成选项，不要和"服务商"耦合（同一服务商可能有多套协议）
- 默认值要**可覆盖**，不要写死

### 4. **Obsidian 插件专属经验**

- 所有 HTTP 请求用 `requestUrl`，绝不用 `fetch`（CORS 只有在桌面版没法 workaround）
- 二进制用 `vault.createBinary`，文本用 `vault.create`
- 设置项改变要 `await this.plugin.saveSettings()`，UI 需要刷新用 `this.display()`
- `Notice` 仅用于通知，错误用 `console.error` 才能在 DevTools 里看到 stack

### 5. **错误信息暴露到终端**

每次调用第三方 API 失败时，一定要把 `response.status + response.text.slice(0, 300)` 抛出来。调试时省 5 倍时间。

## 三、项目启动手册（可复用 Playbook）

这份清单按**执行顺序**列出，每一步都是"跳过会返工"的高价值动作。

### 阶段 0：定义问题（0.5 天）

在动手做任何调研之前，先写清楚这三件事：

- [ ] **用户是谁**：一句话描述目标用户（不要写"所有人"）
- [ ] **核心场景**：用户在什么情境下、想解决什么问题
- [ ] **成功标准**：做到什么程度算 MVP 完成（可验证）

> 例：本项目 = "Obsidian 重度用户把听过的播客从'听过就忘'变成'可搜索的知识资产'，MVP = 粘贴链接就能生成一篇结构化笔记"

### 阶段 1：竞品 & 用户调研（0.5-1 天）

**最前置，最容易跳过，最坑。** 省这一步会导致方向错误，后面越走越远。

- [ ] **同类竞品扫一遍**（至少 3 家，覆盖国内外 + 不同形态）
  - 它们的核心输出是什么（截图留档）
  - 各家独有亮点（哪些值得借鉴）
  - 哪些做不到（我们的差异化机会）
- [ ] **产品定位一句话**：用「A 的 X + B 的 Y + C 的 Z」句式写清楚差异化
- [ ] **明确不做什么**：列出 3 条边界，避免路上偏航

> 本项目因为跳过了这一步，前期没意识到"发言人区分/章节摘要"是刚需，差点做成没有竞争力的产品。

### 阶段 2：技术选型调研（0.5 天）

高返工成本，必须前置。

- [ ] **API 兼容矩阵**（每个要集成的第三方服务，一行一个）

  | 服务商 | 能力 | 协议 | 端点 | 鉴权 | 限流 | 价格 | 关键坑 |
  |--------|------|------|------|------|------|------|--------|

- [ ] **协议差异识别**：哪些服务走 OpenAI 兼容，哪些走 Anthropic，哪些自定义
- [ ] **鉴权方式差异**：Bearer / x-api-key / HMAC / OAuth
- [ ] **同步 vs 异步**：转录类 API 普遍异步（submit + poll），写代码前就要心里有数
- [ ] **测试样本成本估算**：跑通一次 MVP 会烧多少钱

> 本项目因为跳过，踩了：MiniMax 没有转录（413）、MiniMax M2.7 走 Anthropic 协议（404）、火山用 `X-Api-Key` 不是 Bearer。

### 阶段 3：平台 & 技术栈约束（0.5 天）

每个目标平台都有隐藏规则，动手前必须查一遍。

- [ ] **平台能力边界**：插件/扩展能做什么不能做什么（举例：Obsidian 插件不能用 `fetch`，要用 `requestUrl`）
- [ ] **CORS / 权限 / 沙箱**：网络/文件/UI 的限制
- [ ] **API 版本**：目标平台当前稳定版的 API 有哪些（挑未被 deprecated 的）
- [ ] **发布约束**：商店审核规则、依赖限制、打包体积

> 本项目因为跳过，踩了：Obsidian 里 `fetch` 被 CORS 拦截，改 `requestUrl` 返工一轮。

### 阶段 4：架构骨架（0.5 天）

轻量前置，预埋扩展点。

- [ ] **数据模型**：核心 TS interface 先写（`PodcastMeta`、`TranscriptSegment`、`LlmConfig` 等）
- [ ] **目录结构**：按**职责**分而不是按**功能**分
  - ✅ `parsers/` `ai/` `generators/`
  - ❌ `xiaoyuzhou/` `minimax/` `火山/`
- [ ] **抽象层预埋**：即使 MVP 只接一家，也要留 provider 接口
- [ ] **调用图**：画一张"谁调谁"的框图（draw.io 或 Mermaid 即可）
- [ ] **错误流**：想清楚每一步失败时怎么降级/提示

### 阶段 5：MVP 范围切割（0.5 天）

- [ ] **最小可验证链路**：哪些能力是 MVP 必须的，哪些是 v0.2 再做
- [ ] **里程碑拆分**：每个里程碑都应该"能单独跑"
- [ ] **风险排序**：把最不确定的环节放第一个做（先搞定 API 再写 UI）

### 阶段 6：开发前的环境准备（0.5 天）

- [ ] **API Key 全部申请好**：不要写到一半卡在注册账号
- [ ] **调试工具就位**：DevTools、日志、断点
- [ ] **Git 仓库 + .gitignore**：别忘记 key/secret 的忽略
- [ ] **基础测试样本**：准备 1-2 个真实的输入，不要等写完才找

## 四、关于"是否要完美规划再动手"的平衡

**不是所有事都要提前想清楚**，但要区分：

| 类型 | 做法 | 本项目例子 |
|------|------|----------|
| **产品方向**（用户、场景、差异化）| **重度前置**，定错方向全盘返工 | 没做竞品分析差点做成没差异化的产品 |
| **外部依赖**（API、协议、鉴权）| **必须前置**，返工成本高 | Whisper/LLM 两轮重构 |
| **平台约束**（CORS、权限）| **必须前置**，绕不过去 | fetch → requestUrl |
| **架构骨架**（目录、抽象层）| **轻量前置**，预留扩展点 | dispatcher 模式 |
| **内部实现**（算法、UI 细节）| 可以边做边改 | 样式、交互细节 |
| **业务逻辑**（prompt、模板）| 快速迭代，不需要规划 | prompt、Markdown 模板 |

**原则：高返工成本的事前置，低返工成本的事快速试错。**

### 时间预算参考

| 项目规模 | 前置调研时间 | ROI |
|---------|------------|-----|
| 小（1-3 天工作量）| 0.5-1 天 | 省 1-2 天返工 |
| 中（1-2 周）| 1-2 天 | 省 3-5 天返工 |
| 大（1 个月+）| 3-5 天 | 省 1-2 周返工 |

> **反直觉的经验：前置调研花的时间，几乎总是比你以为的多一倍；但它省下的返工时间，也总是比你以为的多一倍。**

## 五、跳过前置步骤的常见借口（和反驳）

| 借口 | 反驳 |
|------|------|
| "这个技术我熟" | 熟的是**你以前用过的那个版本/服务商**，不是**这个新服务商** |
| "先跑通再说" | 跑通一家 ≠ 跑通架构，单 provider 写死后加第二家就是重构 |
| "竞品分析是产品经理的事" | 开发者不做竞品分析 = 闭门造轮子 |
| "时间紧来不及" | 返工比调研贵 3-5 倍时间 |
| "MVP 不用想那么多" | MVP 可以功能少，但**架构不能少**，否则是一次性代码 |

## 五、本项目的遗留技术债

- `llm.ts` 里 OpenAI 和 Anthropic 两个分支内联了，后面如果加第三个协议（Gemini）应该拆 provider
- `main.ts` 越来越长，建议后面把设置 UI 单独抽成 `settings-tab.ts`
- 没有 eval/自动化测试，prompt 改动靠肉眼
- 没有音频缓存策略，同一期节目重试会重复转录（烧钱）

## 六、给未来的自己

> 下次做 BYOK 类产品，先花半天做"API 兼容矩阵"，再花半小时画"provider 抽象"的目录结构，然后再写代码。
> 这一个小时的投入，能省掉两轮重构。
